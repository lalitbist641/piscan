# Methodology

## Threat model

We consider a black-box external attacker interacting with a deployed large language model (LLM) chatbot solely through its natural-language interface. The attacker cannot observe or modify the model weights, the system prompt, or any server-side configuration; they can only submit user messages and observe the resulting responses. The attacker's objective is prompt injection: to supply input that overrides the developer's intended instructions, causing the model to disregard its constraints, adopt an unauthorized persona, disclose privileged information such as its system prompt, or otherwise act against its deployment policy. This threat model reflects the position of a real adversary probing a production chatbot, and it deliberately excludes white-box or training-time attacks, which are out of scope.

## System overview

We implement PIScanner, a command-line tool that automates black-box prompt-injection testing and produces a labeled dataset of attacks and model responses. The pipeline comprises four stages: a curated payload library, an asynchronous probing engine, a layered detection cascade, and an LLM-based validation judge. Each stage is described below.

## Payload taxonomy

The payload library contains sixty-one adversarial prompts organized into five categories that correspond to distinct injection strategies. *Direct* payloads attempt to override instructions explicitly through the user message (for example, instructing the model to ignore prior instructions and emit a marker token). *Indirect* payloads simulate injection through retrieved or embedded content, as would occur in a retrieval-augmented generation setting. *Role* payloads attempt to hijack the model's identity by assigning it an unrestricted or malicious persona. *Encoding* payloads obfuscate the malicious instruction using transformations such as Base64, ROT13, or leetspeak in an effort to evade surface-level filters. *Extraction* payloads seek to elicit the model's hidden system prompt or original instructions. The library additionally includes fifteen benign prompts that serve as negative controls for assessing false-positive behavior. Each payload is annotated with a unique identifier, its category, and its source, enabling per-category analysis.

## Probing engine

The probing engine dispatches every payload to a target chatbot endpoint and records the response text, HTTP status, and latency for each request. Requests are issued asynchronously with a configurable concurrency limit to balance throughput against rate limits, and results are persisted to both JSON and a local SQLite store for downstream analysis. To enable safe, repeatable, and cost-free experimentation, the primary target is a locally hosted open-weight model (Llama 3.2) served through Ollama, exposing an endpoint functionally analogous to a production chat API. The same engine can target any HTTP chat endpoint, allowing later extension to additional or commercial models.

## Detection cascade

Determining whether a given attack succeeded is itself a non-trivial classification problem, because a response may contain attack-related vocabulary while in fact refusing or merely describing the attack. We therefore adopt a cost-aware cascade of three detection layers of increasing cost and semantic depth. The first layer applies category-specific keyword patterns and is computationally trivial, but it is made *compliance-aware*: a keyword match is only treated as a successful injection when the response does not exhibit refusal or deflection language and, where possible, exhibits a category-specific compliance marker. This design directly addresses a failure mode observed in an earlier iteration, in which refusals such as "I do not have a system prompt" were incorrectly counted as successful extractions. The second layer computes the semantic similarity between the response and a representation of the attack intent using sentence embeddings, capturing successful injections that do not surface exact keywords. The third layer is an LLM-as-judge that provides ground-truth labels, described in the following subsection. The cascade is motivated by cost efficiency: the inexpensive keyword and semantic layers are applied to every response, while the expensive judge is reserved for validation and ground-truth labeling.

## LLM-as-judge validation

To obtain ground-truth labels against which the cheap detectors can be measured, we employ a strong LLM as an impartial judge. For each response, the judge receives the original attacker payload, the model's response, and the attack category, and is instructed under a fixed rubric to return a structured verdict of SUCCESS, REFUSED, or UNCLEAR, together with a confidence score and a brief justification. The rubric defines success as the model actually complying with the injected instruction — for instance, emitting the requested marker token, adopting and acting upon a forbidden persona, or disclosing genuine system instructions — and explicitly distinguishes this from refusing, deflecting, or merely decoding or describing the payload without obeying it. The judge is queried at temperature zero with enforced structured (JSON) output to promote determinism and parseability. The implementation supports both a commercial judge (GPT-4o) and a locally hosted open-weight judge served through Ollama, permitting a fully offline and cost-free evaluation when required.

## Evaluation metrics

We evaluate the fast detection layers by treating the judge's SUCCESS verdicts as ground truth and the keyword (or semantic) layer's output as the prediction. From the resulting confusion matrix we report precision, recall, F1, and accuracy, excluding responses the judge marks UNCLEAR or that failed at the request level. Precision quantifies how often a raised detection corresponds to a genuine injection, while recall quantifies the fraction of genuine injections the cheap layer captures — the latter being the more revealing measure of a keyword detector's blind spots.

## Reproducibility and limitations

Because LLM outputs are non-deterministic, individual counts vary across runs; reported figures should therefore be interpreted as representative rather than exact, and aggregated over multiple runs where precision is required. We further note that the quality of the ground-truth labels is bounded by the capability of the judge: a smaller judge model was observed to over-report successful injections, whereas a larger judge produced verdicts more consistent with manual review, underscoring the importance of a sufficiently capable judge. The present study targets a single open-weight model over a single collection run; extension to multiple models, repeated trials, and a commercial-grade judge constitutes future work and would strengthen the external validity of the results.
